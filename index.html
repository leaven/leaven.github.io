Redux概述（一）

发表于 2016-01-09   |  

本文主要阐释基本的redux概念，并基于react实现一些demo，demo部分会用到webpack，es6，本文对此不做介绍，本文适合想要入门redux的同学，不适合已经有实践经验的工程师，本文尽量避免官方文档中的晦涩难懂的概念，根据本人的实际经验与理解进行描述。

Redux是一个思想，并不是react的一个插件，适用于UI基于状态模式的解决方案，而react基于状态的数据流思想完美地契合了redux，所以目前redux大多被用在了react上面。

目的

传统的开发方式一般是View跟Model双向绑定，任何一方的改变都会影响另一方，一个View通常会关联多个Model，一个Model也会关联多个View，往往项目复杂到一定程度后，一个事件监听中会有多个回调，而这些都得你手动去维护，这就是为什么有时候一个需求很小，但是做起来很复杂的原因，精力都被分配到处理以前的逻辑上了。

redux采用了一个reducer的概念，将所有的状态集中在一个地方管理，View层不允许通过dom等操作直接改变state以做到view与data的解耦。

设计思想

redux有三个新概念，action、reducer、store，看个图

官方说法太啰嗦，我简化一下，action是一个函数，代表一个操作，action中必须返回一个字段对动作进行唯一标识，标识的意义也是为了让reducer能够识别每个动作去处理。reducer可以接收action的动作，对state进行操作。
store就是一个大脑，有三个方法分别为getState()、dispatch（action）、subscribe()对应store、action、reducer，getState()用于获取state树， dispatch(action)用于派发action，subscribe()用于监听action执行对应reducer操作。
举个栗子，store就相当于你的老板，action就是你老板制定KPI，每个reducer就是各个团队的经理。
年初你老板会为每个团队制定一条或者几条KPI，制定完之后，相关的团队经理会去执行相关KPI，年末每个团队经理会把执行情况反馈给你老板，这就是一个典型的redux流。公司里只有一个老板，不能有多个，不然会乱了套，经理数量需要根据公司大小去安排，KPI可以制定多个，每个经理可以根据自己的属性或者能力领一个或者多个。

所以一家井井有条的公司运行机制一定是这样的，而公司的组织架构划分则显得尤为重要，这就像redux中的reducer拆分，我们需要站在一个整体的高度上面去抽象去建模，这样才能让组织更高效地运转。

初探React

Why use React

1、面向View层开发，一切皆组件
2、单向数据流
3、VirtualDom，高性能渲染
4、Learn once，write everywhere.(React Native for IOS、React Native for Android)
5、github star数量已破3W

Begin With React

1、require 'react'
2、require 'react-dom'
3、因为react语法需要jsx编译引擎，所以需要引入jsx编译脚本，传送门

现在我们在页面中渲染一个HelloWorld

var HelloWorld = React.createClass({
  render: function() {
    return (
        <div>
          HelloWorld
        </div>
    );
  }
});
ReactDOM.render(
  <HelloWorld />,
  document.getElementById("helloworld")
);

实际效果请戳这里
ReactDOM.render能将JSX模板编译成HTML插入到相应的element中。
注意：
1、自定义的类必须以大写字母开头，如果用小写字母会被认为是标准html元素，导致不能被解析**；
2、return返回必须需要dom标签包裹，否则会不能被解析；
3、JSX 的基本语法规则：遇到 HTML 标签（以 < 开头），就用 HTML 规则解析；遇到代码块（以 { 开头），就用 JavaScript 规则解析。

组件上可以增加任意属性（类似于我们在dom上添加data-*），组件内中可以通过this.props.*获取到组件上的自定义属性。

State

state是React中一个非常重要的概念，所有的UI改变都是通过改变state发生，state改变能够触发UI重绘，所以，所有的交互状态我们都应该尝试去改变state，这一点跟传统的dom交互大不一样，看下面的一个例子

var HelloWorld = React.createClass({
  getInitialState: function() {

    this.loadData();
    return {data: []};
  },
  loadData: function() {
    var me = this;
      $.getJSON(this.props.url, function(data) {
        me.setState({data: data});
      });
  },
  render: function() {
    var person = this.state.data.map(function(d, i) {
      return (
        <div className="name" key={i}>
          name:{d.name}, age:{d.age}
        </div> 
      )
    });
    return (
       <div>
          {person}
       </div>
    );
  }
});
ReactDOM.render(
  <HelloWorld url="./data/demo4.json"/>,
  document.getElementById('example')
);

源码地址
在请求到数据后，我们将state置为获得的data，导致render方法被执行。
注意
1、这里的map会返回一个数组，每一个元素都需要设置一个key，这里建议key设为数组的索引值，因为索引值能保证每个不一样；
2、关于state及props适用场景请参考这里。

获取DOM对象

获取组件内部的dom对象可以通过ref，this.refs是一个dom节点的引用数组，看下面的例子

var InputChange = React.createClass({
    getInitialState: function() {
      return {
        inputValue: ''
      }
    },
    getValue: function() {
      var value = this.refs.nameInput.value;
      this.setState({"name": value});
    },
    render: function() {
      return (
        <div>
          <input type="text" ref="nameInput" onInput={this.getValue} />
          <span>您输入的名字是：{this.state.name}</span>
        </div>
      )
    }
  });
  ReactDOM.render(
    <InputChange />,
    document.getElementById("example")
  );

源码地址
上面代码将input挂到了一个nameInput引用对象，在组件类中通过this.refs.nameInput引用到该input。

组件生命周期

React跟传统的DOM编程不一样，在React中组件的生命状态都在一个order里面，井井有条，什么阶段执行什么方法都由React去完成，所以会比较清晰。
1、componentWillMount
在组件即将render之前立即调用，只会被调用一次
2、componentDidMount
在组件render之后立即调用，只会被调用一次
3、componentWillReceiveProps
在改变组件的props属性时立即调用，在组件初始化的时候不会被调用
4、shouldComponentUpdate
在组件props或者state状态改变之前调用，在组件初始化或者用forceUpdate方法时不会被调用，可以利用这个方法return false即可阻止componentWillUpdate调用
5、componentWillUpdate
在shouldComponentUpdate调用之后立即调用
6、componentDidUpdate
在componentWillUpdate之后刷新DOM完成之后调用
7、componentWillUnmount
在组件被销毁的时候调用

前端开发之观察者模式

什么是观察者模式

观察者模式（有时又被称为发布-订阅Subscribe>模式、模型-视图View>模式、源-收听者Listener>模式或从属者模式）是软件设计模式的一种。在此种模式中，一个目标物件管理所有相依于它的观察者物件，并且在它本身的状态改变时主动发出通知。这通常透过呼叫各观察者所提供的方法来实现。此种模式通常被用来实现事件处理系统。（来源于百度百科）

还是不懂，没关系

今天晚上老大要上线一个活动，这时候PM来通知你了，
“XX，今天有个活动需要你支持一下，巴拉巴拉，上线了发个邮件给我，我先回家了。”

但是没办法，这个事件已经跟你绑定了，只有完成条件才能解绑，于是你疯狂code，终于上完了线，发了邮件通报，这个时候这个事件才算完成。

这个栗子里面，PM就是一个订阅者(bind)，向你订阅了一个邮件通报，他可以不用关心你的code过程，只需要获得通知即可。你就是一个发布者，在满足上线这个要求后，才触发了发邮件的操作（trigger）。

观察者模式适用场景

百度百科里面有个单词大家不造有没有注意到，事件处理，而javascript就是纯事件驱动的语言。为什么说事件驱动适合用观察者模式解决，分析一下事件产生的要素你就知道，一个事件必须包含事件时间、事件对象、事件起因、事件经过、事件结果。

然而以上代码只适用于模板登录情况，如果登录接口是异步接口呢，那只能去login回调函数里面加一个去除广告的逻辑了。事情做到这，你有没有感觉到一丝丝egg hurt，如果PM再加这样的逻辑肿么办？有没有办法做到登录之后就发出通知呢，所有的跟登录相关的事件都绑定到这个通知上去？

仔细分析浏览器事件绑定的原理，我们可以发现，案列一中那个最常见的dom事件绑定代码，其实内部实现原理是这样子的，浏览器为每个节点维护一个eventMap，类似于

eventMap = {
    'click' : [
        fn1,
        fn2
    ],
    'touchstart' : [
        fn3,
        fn4
    ]
}

当浏览器USER Interface线程捕获到用户的click操作时，浏览器就去eventMap里面查找索引为click数据，发现里面有fn1、fn2，并依次执行。

在上面的代码中，我们只是向浏览器订阅了一个click事件，事件在什么时候、什么时间触发全都由浏览器内部实现。在发生click的时候你可以执行fn1、fn2、….fnn。这个不就是我们所需要解决的事件耦合的问题嘛！事情到这，就很简单了，我们只需要实现一个自定义事件队列即可。
参考code，来自alloyteam

Events = function() {

   var listen, log, obj, one, remove, trigger, __this;

   obj = {};

   __this = this;

   listen = function( key, eventfn ) {  

     var stack, _ref;  

     stack = ( _ref = obj[key] ) != null ? _ref : obj[ key ] = [];

     return stack.push( eventfn );

   };

   one = function( key, eventfn ) {

     remove( key );

     return listen( key, eventfn );

   };

   remove = function( key ) {

     var _ref;

     return ( _ref = obj[key] ) != null ? _ref.length = 0 : void 0;

   };

   trigger = function() {  

     var fn, stack, _i, _len, _ref, key;

     key = Array.prototype.shift.call( arguments ); 

     stack = ( _ref = obj[ key ] ) != null ? _ref : obj[ key ] = [];

     for ( _i = 0, _len = stack.length; _i < _len; _i++ ) {

       fn = stack[ _i ];

       if ( fn.apply( __this,  arguments ) === false) {

         return false;

       }

     }

     return {

        listen: listen,

        one: one,

        remove: remove,

        trigger: trigger

     }

   }

现在再来改我们之前的代码

var eventCenter = new Events();

}
